Overview

GitHub is a website where millions of developers gather every day to collaborate on open source software. It’s also the place that hosts billions of lines of code, and also a place where users of software go to report issues they might have. As a developer you can’t avoid using GitHub daily, either to host your code or to make use of other people’s code. This site explains how to use Git from a command line.

Basic git usage

Managing files and Commits

  • git status
    Show the current status for files. The git status command has two sections; the first display modified or deleted files that are currently being tracked by git, and the second shows files not being tracked by git.
  • git add file|directory
    Is used to add one or more files or directories
  • git mv file|directory
    Is used to move one or more files or directories
  • git rm file|directory
    Is used to delete/remove one or more files or directories
  • git commit
    Is used to delete/remove one or more files or directories
  • git commit -m "my message"
    Commit stages adds, moves, and deletes for updates to the current branch. The -m option allows you to create a commit message on the command line. If not included, your configured editor will be started allowing to add a commit message. Include the issue ID addressed by this change in the commit message.
  • git commit --amend
    Use the --amend option to add the current changes to a prior commit
Below is an example of checking git status, adding a file, and then making a commit using the command line option -m. Note, you should include the issue the commit is related to in the commit message.

Working with branches
See what branches exists
Below is a list of commonly used commands for seeing what git branches are available. This example uses the PavedRoad GitHub pages as an example. Examples of all three commands can are directly below.
HEAD is the reference to the latest commit in the current checked out branch. It usually points to a branch called "master," but it can point to any branch.
  • git branch
    Shows a list branches current on your machine
  • git branch -r
    Shows all branches on the remote server, aka "origin" in most cases
  • git branch -a
    Shows all local and remote branches
  • git fetch
    Add remote branches to your local ones
* represents the current branch 
$ git branch

*  gh-pages
  master


$ git branch -r

  origin/HEAD -> origin/master
  origin/gh-pages
  origin/master


$ git branch -a

* gh-pages
  master
  remotes/origin/HEAD -> origin/master
  remotes/origin/gh-pages
  remotes/origin/master
Creating new branches

The command git checkout is used to create or switch branches.

  • git checkout -b mynewbranch
    Creates a new branch off of the HEAD/master
  • git checkout -b mynewbranch anotherbranch
    Creates a new branch off of another branch.
  • git stash
    If you have uncommitted changes, git will not let you switch/ create a new branch. If you want to keep your changes, you can use the "stash" command to save them.
  • git stash list
    Use the list option to see a list of saved, aka stashed, changes.
  • git stash apply stashid
    The apply option applies the saved change to your current working branch.
  • git stash drop stashid
    The drop option deletes a saved change.

Merging branches
In our examples, we have a branch called "gh-pages" and one called "git-content". The "git-content" branch contains this document. The "gh-pages" branch are changes stagged for the master branch. If we want the changes in our "git-content" branch to be merged with "gh-pages", we use the git merge command.
You can merge from one branch into another multiple time.
  • start your merge by switching to the march you want to merge into using the git checkout command

Merging conflicts
A merge conflict happens when the changes in one branch for the same section of the code are different. Git doesn't know which version is correct so, it is up to the developer to tell git which changes to keep.
We will demonstrate this by creating a conflict between the "gh-pages" branch and the "git-content" branch. The example below demonstrates:

Pushing changes and Pulling updates from the origin
So far we have made local changes, what happens when we want to push our changes to the origin so other developers can see them, or pull in other developers changes?
The following commands accomplish this:
  • git push -u origin mybranch
    Push my changes to origin, use the -u option to update the origin
  • git push origin mybranch
    Push my changes to origin
  • git pull origin mybranch
    Pull other developers changes from the origin
The following example demonstrates an initial push with the -u. Note: GitHub suggests creating a pull request; you should use the link provided to do that.

Seeing git hisotry with git log
To see a history of all your commits, use the command "git log". The log command has many different options to control the level of verbosity and output format.
  • git log -n 10 --since="2016-06-24"
    Displays the commit id, date, author, and commit message. Limit the number of enteries returned using the -n option or --sence and a date.
  • git log -n 10 --oneline
    Reduce output to one line that includes the commit id and commit message.
  • git log --oneline --decorate --graph --all -n 10
    The graph option provides a graphical history of branches, commits, and merges.
