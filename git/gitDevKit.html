Basic git usage

Managing files and Commits

  • git status
    Show the current status for files. This will display files that have been modifed, and files not currently being tracked by git.
  • git add file|directory Is used to add one or more files or directories
  • git mv file|directory Is used to move one or more files or directories
  • git rm file|directory Is used to delete/remove one or more files or directories
  • git commit
    Is used to delete/remove one or more files or directories
  • git commit -m "my message"
    Commits/stagges add/mv/deletes for upload to the origin -m option allows you to create a commit message on the command line
  • git commit --amend
    Use the --amend option to add the current changes to a prior commit

Working with branches
See what branches exists

Below is a list of commonly used commands for seeing what git branches are available. This example uses the PavedRoad GitHub pages as an example. Examples of all three commands can be found directly below.

HEAD is the reference to the lastest commit in the current checked out branch. Noramlly it points to a branch called "master", but it can point to any branch.

  • git branch Shows current local, aka on your machine branches
  • git branch -r Shows all branches on the remote server, aka "origin" in most cases
  • git branch -a Shows all local and remote branches
  • git fetch Add remote branches to your local ones
* represents the current branch 
git branch

*  gh-pages
  master


git branch -r

  origin/HEAD -> origin/master
  origin/gh-pages
  origin/master


git branch -a

* gh-pages
  master
  remotes/origin/HEAD -> origin/master
  remotes/origin/gh-pages
  remotes/origin/master
Creating new branches

The command git checkout is used to create and switch branches.

  • git checkout -b mynewbranch Creates a new branch off of the HEAD/master
  • git checkout -b mynewbranch anotherbranch Shows all branches on the remote server, aka "origin" in most cases
  • git stash If you have uncommitted changes, git will not let you switch/ create a new branch. In this case, you can use the "stash" command to save your changes and then switch branches.
  • git stash list Shows all branches on the remote server, aka "origin" in most cases
  • git stash apply stashid Shows all branches on the remote server, aka "origin" in most cases
  • git stash drop stashid Shows all branches on the remote server, aka "origin" in most cases

Merging branches
In our examples, we have a branch called "gh-pages" and one called "git-content". The "git-content" branch contains this document. The "gh-pages" branch are changes being stagged for the master branch. If we want the changes in our "git-content" branch to be merged with "gh-pages", we us the git merge command.
You can merge from one branch into another multiple times. This enables bug fixes from a feature branch to be merged/updated or changes to be gradually added.
  • start your merge by switching to the march you want to merge into using the git checkout command
# Change branches
$ git checkout gh-pages

Switched to branch 'gh-pages'

$ git merge git-content

Merging conflicts
A merge conflict happens when the changes in one branch for the same section of code are different. Git doesn't know which version is correct so it is up to the developer to tell get which changes to keep.
We will demonstrate this by creating a conflict between the "gh-pages" branch and the "git-content" branch. The example below demonstrates:
  • switch to the gh-pages branch and changing "git" to "get" on a line of code

Pushing changes and Pulling updates from the origin
So far we have made local changes, what happens when we want to push our changes to the origin so other developers can see them, or pull in other developers changes?
The following commands accomplish this:
  • git push -u origin mybranch push my changes to origin, use the -u option to update the origin
  • git push origin mybranch push my changes to origin
  • git pull origin mybranch pull other developers changes from the origin
The following example demonstrate and initial push with the -u. Note: GitHub suggests greating a pull request, you should use the link provided to do that.
